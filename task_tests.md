# Задача: Покрытие проекта Автотестами (Pytest + Mocks)

## Контекст
Нам нужно покрыть написанный бэкенд (FastAPI) автотестами. Тесты должны работать изолированно, быстро и НЕ требовать реальных ключей Anthropic, реального подключения к SMTP или запуска реального браузера Playwright.

## Требования к тестированию:
1. **Зависимости:** Добавь `pytest`, `pytest-asyncio`, `httpx` (для TestClient) и `pytest-mock` в зависимости.
2. **Структура:** Создай папку `tests/` в корне проекта.
3. **Тестовая База Данных:** Настрой фикстуру (fixture) в `conftest.py`, которая будет создавать in-memory SQLite базу (или временную схему PostgreSQL) для каждого теста, чтобы не мусорить в боевую БД.
4. **Моки (Mocks) внешних сервисов (КРИТИЧНО):**
   - ЗАМОКАЙ вызовы к Anthropic API. Тест должен думать, что Клод вернул текст письма, не делая реального сетевого запроса.
   - ЗАМОКАЙ SMTP сервер. Мы не должны отправлять реальные письма при запуске тестов.
   - ЗАМОКАЙ Playwright. Для тестирования эндпоинта парсера, замокай `scraper_service.py`, чтобы он просто возвращал фейковый список из 2-3 компаний.

## Какие тесты написать:
1. `test_auth.py`: Проверка успешного и неуспешного логина админа.
2. `test_leads.py`: Проверка загрузки лидов (имитация загрузки CSV) и создания кампании.
3. `test_email_engine.py`: Проверка фоновой задачи генерации письма. Убедиться, что статус лида меняется на `contacted`, а мок SMTP вызывается 1 раз.
4. `test_parser_endpoint.py`: Проверка того, что эндпоинт `/api/parser/run` принимает валидные данные и успешно запускает замоканную фоновую задачу.

Напиши конфигурацию `pytest.ini`, `conftest.py` и сами файлы тестов. Предоставь инструкцию, как запустить тесты в терминале.
